# Strategy Risk Layer and Telegram Risk Report Design

## Context

Production paper-mode is stable and producing fills, but the current behavior has two gaps:

1. The strategy can accumulate large directional inventory on a single market.
2. The Telegram report summarizes PnL but does not expose enough risk context to decide whether the bot can keep running safely.

Recent production observations:

- Mode: paper trading.
- Container uptime since last restart: ~3 days.
- Paper fills: 145.
- Notional volume: ~$174.09.
- Logged cumulative realized PnL: +$4.92.
- Estimated fair-based unrealized PnL: +$10.99.
- Net inventory: approximately SHORT 167 contracts on the primary market.
- Quote concentration: ~99.95% on one market.

The goal is to make the bot safer in paper-mode now, while designing the same risk layer so it can later gate `small_live` trading. Live trading remains disabled.

## Goals

1. Prevent uncontrolled inventory growth.
2. Add automatic reduce-only behavior when inventory limits are approached or breached.
3. Track trading activity accurately: fills, contracts, notional, positions, and markets.
4. Expose inventory, concentration, worst-case loss, and valuation mode in Telegram reports.
5. Compute an explicit risk status: `OK`, `WATCH`, `WARNING`, or `CRITICAL`.
6. Keep the design reusable by `run-paper.ts`, `run-shadow.ts`, and future `run-live.ts`.

## Non-Goals

- Do not enable live trading.
- Do not place real orders.
- Do not implement full orderbook-depth liquidation simulation in the first pass.
- Do not build a database or historical analytics service.
- Do not refactor unrelated strategy code.

## Architecture

Add three focused units:

```text
src/risk/strategy-risk-manager.ts
src/accounting/trading-activity-tracker.ts
src/reporting/telegram-risk-report.ts
```

### `StrategyRiskManager`

Responsible for turning positions, market activity, and latest book/fair data into risk decisions.

Inputs:

- current positions from `PaperPnlTracker`;
- latest books;
- latest fair prices;
- per-market quote/fill activity;
- configured risk and inventory limits;
- kill switch state where available.

Outputs:

```ts
type RiskStatus = 'OK' | 'WATCH' | 'WARNING' | 'CRITICAL';

interface MarketRiskDecision {
  conditionId: string;
  tokenId: string;
  riskStatus: RiskStatus;
  reasons: string[];
  reduceOnly: boolean;
  allowBuy: boolean;
  allowSell: boolean;
  inventoryUsagePct: number | null;
  netPosition: number;
  positionSide: 'LONG' | 'SHORT' | 'FLAT';
  avgEntryPrice: number | null;
  currentFair: number | null;
  currentBid: number | null;
  currentAsk: number | null;
  fairUnrealizedPnl: number;
  exitPnlAtBestBidAsk: number | null;
  worstCaseLossToZero: number | null;
  worstCaseLossToOne: number | null;
}
```

Behavior:

- If position is SHORT and exposure is above reduce-only threshold, block new SELL quotes and allow BUY quotes.
- If position is LONG and exposure is above reduce-only threshold, block new BUY quotes and allow SELL quotes.
- If exposure is below soft threshold, allow both sides.
- If inventory exceeds hard threshold, risk status becomes at least `CRITICAL` for that market.
- If single-market concentration exceeds 90%, report at least `WARNING`; in live-mode this should become `CRITICAL`.
- If book/fair data is stale while active quotes exist, report `CRITICAL` and block inventory-increasing quotes.

Initial thresholds should come from existing typed config where possible:

- `inventory.softLimitPct`
- `inventory.hardLimitPct`
- `inventory.maxMarketExposureUsd`
- `inventory.maxTotalStrategyExposureUsd`

If a required limit is ambiguous, the first implementation should make it explicit in config instead of relying on `env.maxExposureUsd / 100`.

### `TradingActivityTracker`

Responsible for counting trading activity without overloading `PaperPnlTracker`.

Tracks:

```ts
interface TradingActivitySnapshot {
  fillsTotal: number;
  buyFills: number;
  sellFills: number;
  buyContracts: number;
  sellContracts: number;
  totalContracts: number;
  buyNotional: number;
  sellNotional: number;
  notionalVolume: number;
  avgFillPrice: number | null;
  quoteTraces: number;
  quoteGeneratedCount: number;
  quoteRejectedCount: number;
  activeMarkets: number;
  primaryMarketConditionId: string | null;
  primaryMarketQuoteTraces: number;
  primaryMarketQuoteSharePct: number | null;
}
```

Rules:

- `fillsTotal` counts fill events, not positions.
- `totalContracts` sums filled contract size.
- `notionalVolume` sums `filledPrice * filledSize`.
- Quote traces count quote decisions generated by the strategy.
- Rejected/skipped quotes should be counted when risk manager blocks a side.

This separates three concepts:

- fills: execution events;
- positions: open inventory;
- markets: venues where the bot quoted or filled.

### `TelegramRiskReport`

Responsible only for formatting report input into Telegram HTML text.

Inputs:

```ts
interface TelegramRiskReportInput {
  mode: 'paper' | 'shadow' | 'small_live' | 'disabled';
  startedAt: Date;
  reportAt: Date;
  warningsCount: number;
  errorsCount: number;
  pnl: {
    realizedPeriod: number;
    realizedCumulative: number;
    unrealizedFairBased: number;
    estimatedMakerRebate: number;
    estimatedTotalPnl: number;
    valuationMode: 'fair' | 'bid_ask' | 'orderbook_depth';
  };
  activity: TradingActivitySnapshot;
  risk: {
    status: RiskStatus;
    reasons: string[];
    reduceOnlyActive: boolean;
    killSwitchActive: boolean;
    topMarketDecision: MarketRiskDecision | null;
    singleMarketConcentrationPct: number | null;
    unrealizedToRealizedRatio: number | null;
  };
  marketTitleByConditionId: Map<string, string>;
}
```

Output sections:

1. Header and global status.
2. Health: mode, app uptime, warnings/errors.
3. PnL: realized period, realized cumulative, unrealized, estimated rebates, estimated total, valuation mode.
4. Activity: fills, BUY/SELL contracts, notional, quote traces.
5. Inventory: side, net position, average entry, current fair/bid/ask, inventory usage.
6. Risk: reasons, concentration, worst-case loss, reduce-only, kill switch.
7. Main market and action recommendation.

The report must stop using `Total Trades` for position count. It should use `Fills`, `Open Positions`, and `Active Markets` explicitly.

## Strategy Data Flow

For each market evaluation:

```text
latest books + positions + activity
  -> compute fair price
  -> risk manager evaluates current market
  -> quote candidate generation
  -> risk manager validates side
  -> submit paper order or skip/reject quote
  -> activity tracker records quote generated/rejected
  -> fills update pnl tracker and activity tracker
```

Risk must gate the quote before order submission, even in paper-mode. This lets paper-mode test live-ready behavior.

## PnL and Valuation Rules

### Realized PnL

Use `PaperPnlTracker` realized PnL.

Report both:

- period realized PnL from scheduled report window;
- cumulative realized PnL from process start.

### Unrealized PnL

First implementation uses fair-based unrealized PnL:

- LONG: `netPosition * (currentFair - avgEntryPrice)`.
- SHORT: `abs(netPosition) * (avgEntryPrice - currentFair)`.

The report must label this as fair-based.

### Exit PnL at best bid/ask

First implementation should compute a simple best-level estimate when bid/ask is available:

- LONG exit value uses best bid.
- SHORT exit cost uses best ask.

If data is unavailable, report `not available` instead of pretending fair value is exit value.

### Worst-case loss

- LONG to zero: `netPosition * avgEntryPrice`.
- SHORT to one: `abs(netPosition) * (1 - avgEntryPrice)`.

## Risk Status Rules

Global status is the maximum severity among active market decisions and process health signals.

Recommended initial rules:

### OK

- no warnings/errors;
- inventory usage below 50%;
- concentration below 75%;
- no stale fair/book with active quotes.

### WATCH

- inventory usage 50–70%; or
- concentration 75–90%; or
- unrealized/realized ratio between 2x and 3x.

### WARNING

- inventory usage 70–90%; or
- concentration above 90%; or
- unrealized/realized ratio above 3x; or
- high single-market exposure in paper-mode.

### CRITICAL

- inventory usage above 90%; or
- hard inventory limit breached; or
- stale fair/book with active quotes; or
- kill switch active; or
- live-mode has unexpected restart, errors affecting cancel/order placement, or disabled risk controls.

In paper-mode, high concentration can be `WARNING` unless combined with inventory hard-limit breach. In live-mode, high concentration above 90% should be `CRITICAL`.

## Error and Health Tracking

The app should track process-level health internally:

- `startedAt` from process startup;
- warnings count;
- errors count;
- last heartbeat/report time.

Docker container status and Docker restart count are not available from inside the bot unless the runtime provides them. The first implementation should report app uptime, not Docker uptime. Docker-level reporting can be added later via an external supervisor.

## Testing Strategy

Unit tests:

1. `StrategyRiskManager`
   - blocks SELL when short above reduce-only threshold;
   - blocks BUY when long above reduce-only threshold;
   - allows both sides below soft threshold;
   - computes worst-case loss for long and short;
   - escalates status for concentration and hard inventory breach.

2. `TradingActivityTracker`
   - counts fills accurately;
   - separates fills from positions;
   - computes contracts, notional, and average fill price;
   - computes primary market quote share.

3. `TelegramRiskReport`
   - includes required sections;
   - does not include misleading `Total Trades` label;
   - labels valuation mode;
   - includes risk status and reasons.

Integration test:

- Simulate a market with large short inventory.
- Verify SELL quote is rejected, BUY quote remains allowed.
- Verify Telegram report shows reduce-only and risk warning/critical status.

## Rollout Plan

1. Add activity tracking and fix misleading report labels.
2. Add strategy risk manager with tests.
3. Gate paper quotes through risk manager.
4. Add risk-aware Telegram report.
5. Run test suite.
6. Deploy to production paper-mode only.
7. Observe at least 2–3 scheduled report cycles before considering any live-readiness work.

## Live-Readiness Gate

Do not enable live trading until Telegram reports expose and strategy code enforces:

- inventory caps;
- reduce-only mode;
- kill switch status;
- bid/ask exit valuation;
- worst-case loss;
- market concentration;
- confirmed fees/rebates where available;
- errors affecting order placement/cancel.

## Acceptance Criteria

- Telegram no longer reports `Total Trades: 1` for many fills.
- Report includes fills, contracts, notional, open positions, active markets, and quote traces.
- Report includes period and cumulative realized PnL.
- Report labels unrealized PnL as fair-based unless bid/ask exit valuation is used.
- Report includes inventory side, net size, average entry, current fair, and worst-case loss.
- Strategy blocks inventory-increasing quotes when reduce-only is active.
- Paper-mode continues to run without real orders.
- Tests cover risk decisions, activity counting, and report formatting.
